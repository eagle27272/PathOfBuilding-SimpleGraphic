## Instructions to build SimpleGraphic

The directory names and locations chosen for your environment are flexible, below we use the following:

* source directory - `PoB-SimpleGraphic`
* build directory - `build-SimpleGraphic`
* Path of Building dev tree - `PoB`
* install directory - `PoB/runtime/<platform>-<architecture>` for native PathOfBuilding-PoE2 runtimes, or `PoB\runtime` for the legacy Windows runtime

Try to avoid paths with spaces, some of our dependencies are brittle and won't build properly then.

1. Clone the repository with `git clone --recursive` to obtain the sources and the submodules for dependencies, like:
    ```powershell
    git clone --recursive "https://github.com/eagle27272/PathOfBuilding-SimpleGraphic" "PoB-SimpleGraphic"
    ```

2. For a release-style native runtime archive, run the package script from the
   source directory:
    ```sh
    SIMPLEGRAPHIC_RUNTIME_TARGET=macos-arm64 ./scripts/package-runtime.sh
    ```

   The script detects the host target by default, bootstraps vcpkg when needed,
   installs a flat runtime tree, and writes
   `SimpleGraphicRuntime-<platform>-<architecture>.tar`. Set
   `SIMPLEGRAPHIC_VCPKG_TRIPLET` when building a target without a built-in
   default triplet. For a future macOS CPU name, set
   `SIMPLEGRAPHIC_CMAKE_OSX_ARCHITECTURES` if CMake needs a different
   architecture spelling than the runtime target name. For future platform
   labels whose installed file names do not match the Windows/macOS/Linux
   defaults, set `SIMPLEGRAPHIC_ENTRY_LIBRARY`, `SIMPLEGRAPHIC_LUA_MODULE_EXT`,
   or the per-module overrides `SIMPLEGRAPHIC_LCURL_MODULE`,
   `SIMPLEGRAPHIC_LUA_UTF8_MODULE`, `SIMPLEGRAPHIC_SOCKET_MODULE`, and
   `SIMPLEGRAPHIC_LZIP_MODULE`. The package script also writes
   `SimpleGraphicRuntime.json` into the archive and validates the archive
   contract before reporting success. If you package from the same checkout in
   multiple host environments, set `SIMPLEGRAPHIC_VCPKG_ROOT` to a vcpkg tree
   for the current host so a generated vcpkg binary from macOS, Linux, or
   Windows is not reused by the wrong OS.

   Set `SIMPLEGRAPHIC_DRY_RUN=1` with the same target variables to print the
   resolved runtime target, vcpkg triplet, CMake generator settings, archive
   path, entry library, and Lua module names without bootstrapping vcpkg or
   running the build. Use this when checking a new CI target before spending
   time on a full package build.

   CI also publishes `SimpleGraphicRuntime-index.json`, generated by
   `scripts/write-runtime-index.py`, alongside the runtime archives. The index
   records each target's archive name, normalized platform and architecture,
   size, SHA-256 hash, entry library, entry points, and Lua module names for
   PathOfBuilding-PoE2 release ingestion. The runtime workflow can be run
   manually with a blank `runtime_target` input to rebuild the full release
   matrix, or with `runtime_target`, `vcpkg_triplet`, runner, and CMake inputs
   to package and validate one future target before adding it to the release
   matrix.

   On Ubuntu Linux, install the same native build prerequisites as the release
   workflow before packaging:
    ```sh
    sudo apt-get install -y libgl1-mesa-dev libwayland-dev libx11-dev \
      libx11-xcb-dev libxcursor-dev libxext-dev libxi-dev libxinerama-dev \
      libxkbcommon-dev libxrandr-dev mesa-common-dev pkg-config python3-venv \
      wayland-protocols
    ```

3. To configure a build directory manually for debugging or IDE use, on Windows
   with Visual Studio x64:
    ```powershell
    cmake -B "build-SimpleGraphic" -S "PoB-SimpleGraphic" -A x64 -G "Visual Studio 17 2022" --toolchain "vcpkg\scripts\buildsystems\vcpkg.cmake" -DCMAKE_INSTALL_PREFIX="PoB\runtime"
    ```

   For Windows ARM64, use the same generator with `-A ARM64`,
   `SIMPLEGRAPHIC_RUNTIME_TARGET=win32-arm64`, and the `arm64-windows`
   vcpkg triplet when packaging. Build this target on a native Windows ARM64
   runner or host because LuaJIT's MSVC build runs generated ARM64 tools during
   the package build.

   On macOS or Linux, choose the same platform-architecture target name that
   PathOfBuilding-PoE2 uses, such as `macos-arm64`, `macos-x64`,
   `linux-arm64`, or `linux-x64`. Prefer the matching dynamic vcpkg triplet
   (`arm64-osx-dynamic`, `x64-osx-dynamic`, `arm64-linux-dynamic`,
   `x64-linux-dynamic`) when creating distributable runtime archives:
    ```sh
    cmake -B build-SimpleGraphic -S PoB-SimpleGraphic \
      -DCMAKE_BUILD_TYPE=Release \
      -DCMAKE_TOOLCHAIN_FILE=PoB-SimpleGraphic/vcpkg/scripts/buildsystems/vcpkg.cmake \
      -DVCPKG_TARGET_TRIPLET=arm64-osx-dynamic \
      -DCMAKE_INSTALL_PREFIX=PoB/runtime/macos-arm64
    ```

4. On Windows, open the generated `build-SimpleGraphic\SimpleGraphic.sln` file in Visual Studio if you want to build or debug from the IDE.

5. There are three primary build configurations of interest for development largely aligning with the CMake configurations of the same name:

    * `Debug`: Low optimization, easy to step through logic and inspect values without it being all optimized out;
    * `RelWithDebInfo`: Medium optimization, still somewhat debuggable but also better performance;
    * `Release`:  Full optimization - modified from the stock CMake configuration to still generate debug symbols.

6. Build the project to compile the dependencies and SimpleGraphic itself:
    ```sh
    cmake --build build-SimpleGraphic --config Release
    ```

7. Install the runtime files to the install directory:
    ```sh
    cmake --install build-SimpleGraphic --config Release
    ```

The install tree is intentionally flat. It contains `SimpleGraphic.dll` on
Windows, `libSimpleGraphic.dylib` on macOS, or `libSimpleGraphic.so` on Linux,
plus the Lua C modules (`lcurl`, `socket`, `lua-utf8`, `lzip`) and shared
runtime dependencies needed by the native launcher. The package script clears
the install directory before installing so stale files cannot be captured in the
manifest or archive. New platforms outside the supported release matrix should
record their actual entry library and Lua module file names in
`SimpleGraphicRuntime.json`; the verifier treats those manifest fields as the
archive contract for unknown platform labels while keeping strict names for
Windows, Linux, and macOS. The manifest also records the full flat file list
owned by the SimpleGraphic payload so consumers can update SimpleGraphic files
without removing native launcher files from the same target directory.

To validate an archive without rebuilding it, run:

```sh
scripts/verify-runtime-archive.py SimpleGraphicRuntime-macos-arm64.tar
```

To validate a custom artifact set before a future target is part of the release
matrix outside GitHub Actions, run the release validator with explicit expected
targets:

```sh
SIMPLEGRAPHIC_EXPECTED_RUNTIME_TARGETS=freebsd-riscv64 \
SIMPLEGRAPHIC_REQUIRE_LEGACY_WINDOWS_ARCHIVE=0 \
scripts/validate-runtime-artifacts.sh runtime-artifacts
```

To smoke-test a host-compatible archive by loading the packaged entry library
and running a tiny `Exit()` Lua script through `RunLuaFileAsConsole`, run:

```sh
scripts/smoke-runtime-archive.py SimpleGraphicRuntime-macos-arm64.tar
```

If the archive was cross-built for a different CPU architecture, pass
`--allow-incompatible-host` to validate extraction while leaving entrypoint
execution to a matching CI runner.

The verifier checks for safe tar paths, a flat layout, matching
`SimpleGraphicRuntime.json` metadata, a file ownership list that matches the
archive contents, the required SimpleGraphic entry library, the Lua C modules
expected by PathOfBuilding-PoE2, the required SimpleGraphic entry exports, the
architecture of every bundled native binary, and relocatable native dependency
paths. Linux archives must use `$ORIGIN` RPATH/RUNPATH entries, and macOS
archives must use local `@loader_path`/`@rpath` style install names instead of
build-machine absolute paths. The archive must also omit macOS metadata sidecar
files and include every non-system library referenced by its native binaries.


## Debugging SimpleGraphic

The debugger settings in Visual Studio can't quite be communicated through CMake scripts so we have to do some manual setup to point out which executable and what working directory and command line arguments we wish to use to debug the shared library.

1. In Visual Studio open the Project Properties for the SimpleGraphic project.

2. Go to the Debugging page and set the following fields, either for each configuration or once for "All Configurations":

    * "Debugger to launch" - `Local Windows Debugger`
    * "Command" - `$(ProjectDir)..\PoB\runtime\Path{space}of{space}Building.exe`
    * "Command Arguments" - leave this empty unless you wish to launch a specific Lua entrypoint or test something like URLs on the command line
    * "Working Directory" - `$(ProjectDir)..\PoB\runtime`

3. This debugger configuration will use the shared library that is in the PoB install directory, so remember to run the install step before debugging to deploy the library and its dependencies.

4. Set breakpoints and launch new debug sessions from inside Visual Studio or attach to an existing process, as long as the sources match the debug information in the shared library it'll all work fine.


## Development tricks

A PowerShell script like the following can be run with Administrator rights to link a Windows DLL file in the build directory into a PoB runtime tree to avoid having to run the install step for every change:

```powershell
param ($Config="RelWithDebInfo")
$BuildTree = "..\..\build-SimpleGraphic"
$BuildDir = "${BuildTree}\$Config"
$RunDir = ".\PoB\runtime"
$VcpkgBinDir = "${BuildTree}\vcpkg_installed\x64-windows\bin"

$Suffix = ""
if ($Config -eq "Debug")
{
	$Suffix = "d"
}

New-Item -ItemType SymbolicLink -Force -Path "${RunDir}\SimpleGraphic.dll" -Target "${BuildDir}\SimpleGraphic.dll"
```

Don't forget to re-run the script if you change the configuration so that the correct Windows DLL is used by the application.
